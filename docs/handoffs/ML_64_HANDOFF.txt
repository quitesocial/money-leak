ML-64 Backup/Sync Foundation Layer v1 Handoff

Summary
- Added the hidden backup/sync foundation layer for future manual backup.
- Added Supabase remote transaction/category backup table DDL with RLS.
- Added narrow TypeScript sync contracts, a local read boundary, a manual-call
  backup service skeleton, and a fake in-memory adapter for tests.
- The app remains local-first. Guest/local mode remains permanently available.
- No production backup button, restore UI, automatic sync, background sync,
  remote merge, delete account, Apple Sign-In, login wall, or login-triggered
  upload was added.

Migration
- Added supabase/migrations/20260520010000_create_remote_backup_tables.sql.
- Creates only public.remote_transactions and public.remote_categories.
- Both tables use user_id uuid references auth.users(id) on delete cascade.
- Both tables use composite primary keys: (user_id, id).
- Both tables store remote timestamps as timestamptz.
- Both tables include deleted_at for future soft-delete-compatible backup/sync.
- Both tables enable Row Level Security.
- Anonymous/public access is revoked.
- Authenticated users can select, insert, update, and delete only their own rows
  through auth.uid() ownership policies.
- Physical delete is allowed by policy for future account/privacy/manual cleanup
  needs, but ML-64 app code does not call physical delete.
- No remote_settings or sync_metadata table was created.

App changes
- Added src/lib/sync/sync-types.ts.
- Added src/lib/sync/local-backup-data-source.ts.
- Added src/lib/sync/backup-service.ts.
- Added src/lib/sync/fake-remote-backup-adapter.ts.
- Backup payload mapping converts local epoch milliseconds to ISO strings for
  remote timestamptz writes.
- Local transaction category is mapped to remote categoryId.
- Normal backup payloads include active transactions and non-deleted categories.
- Archived categories are included because archive is product state and keeps
  transaction references intact.
- Normal ML-64 payloads do not include tombstones.
- Backup stays disabled by the existing feature flag default.
- Backup skips safely in guest mode and requires a non-empty authenticated user
  id before adapter writes.
- No real Supabase backup adapter was added or wired.
- No Settings, auth-store, router, Google Auth, Supabase config, redirect, or
  Account diagnostics UI code was reworked.

Tests
- Added src/lib/sync/__tests__/backup-service.test.ts.
- Covered local read boundary filtering, local-to-remote payload mapping,
  category-to-categoryId mapping, required authenticated user id, guest-mode
  skip behavior, disabled feature flag behavior, non-mutating payload prep,
  deleted row exclusion, archived category inclusion, fake adapter idempotent
  upserts by (userId, id), and safe recoverable local/remote failures.
- Tests use a fake adapter only and make no live Supabase network calls.
