ML-69 Delete Account / Privacy Controls v1 Handoff

Summary
- Added authenticated-only Delete Account / Privacy controls in Settings.
- Delete Account deletes app-owned cloud data for the current authenticated user
  and then signs out through the existing auth store flow.
- Local SQLite transactions/categories remain untouched and visible after the
  app returns to guest/local mode.
- Guest/local mode remains permanently available. There is no login wall.

Implementation notes
- Added a narrow delete account service boundary under src/lib/account.
- The Supabase adapter uses the existing anon client and authenticated RLS.
- Remote deletion order is remote_transactions by user_id, remote_categories by
  user_id, then profiles by id.
- The mobile app does not use or introduce a Supabase service-role key.
- Settings shows Delete Account only for authenticated users, separate from Sign
  Out.
- Delete Account confirmation states that cloud account/backup data will be
  deleted and local device data will stay.
- Failure UI is generic: "Couldn't delete account. Try again."
- Delete Account does not call backup, restore, import, export, transaction
  load, category load, or local write actions.

Supabase scope
- ML-69 deletes app-owned rows from public.remote_transactions,
  public.remote_categories, and public.profiles.
- Full Supabase Auth auth.users hard-delete is out of scope for ML-69 because
  it requires an Edge Function or another server-side admin/service-role path.
- The new migration grants authenticated profile owner-delete access and adds
  the profiles_delete_own RLS policy.

Manual QA checklist
- Log in through Google or Apple.
- Create or confirm local transactions/categories are visible.
- Create a manual backup so remote rows exist for the account.
- Open Settings and confirm Privacy/Delete Account are visible only while
  authenticated.
- Tap Delete Account, cancel the confirmation, and confirm no remote delete or
  sign out occurs.
- Tap Delete Account again, confirm deletion, and verify remote_transactions,
  remote_categories, and profiles rows for that user are deleted.
- Confirm the app signs out and returns to guest/local mode.
- Confirm local transactions/categories remain visible after deletion.
- Confirm Sign Out remains separate and unchanged.
- Confirm guest mode shows no Delete Account UI.
- Confirm no backup, restore, sync, upload, import, export, local delete, or
  local unlink runs as part of Delete Account.

Non-goals
- No Supabase Auth auth.users hard-delete from the mobile app.
- No service-role usage in mobile app code.
- No Google Auth, Apple Auth, Supabase config/env, backup, restore, or local
  account-linking rework.
- No local SQLite transaction/category deletion or mutation.
- No automatic backup, restore, sync, or remote merge.
- No database/store/domain-model expansion beyond the account delete boundary.
- No CSV, tab navigation, pushed route, or styling-system changes.
