Final status

Product and architecture outcome
- Balance is now a first-class backup/restore/manual-sync domain.
- Balance additions are still not Transactions.
- Transaction CSV v1 remains unchanged:
  id,amount,category,isLeak,leakReason,note,createdAt
- Current balance remains:
  sum(active balance entries) - sum(active/non-deleted expense transactions)
- Home History now displays balance additions and expense transactions together
  in one chronological feed.
- No balance edit/delete UI was added.
- No new sync trigger was added.

Remote Supabase changes
- Added migration:
  supabase/migrations/20260529000000_create_remote_balance_tables.sql
- New remote tables:
  public.remote_balance_types
  public.remote_balance_entries
- Both tables use:
  user_id uuid references auth.users(id) on delete cascade
  primary key (user_id, id)
  schema_version integer
  source_device_id text nullable
  created_at / updated_at timestamptz
  deleted_at timestamptz nullable
- remote_balance_entries.amount uses double precision.
- RLS is enabled.
- public/anon access is revoked.
- authenticated users get own-row CRUD policies only.
- The mobile app still uses only the Supabase anon client.
- Delete-account Edge Function cleanup now deletes remote balance entries, then
  remote balance types, before existing app-owned rows.

Local Balance changes
- Balance types and entries now include sync metadata:
  ownerId
  updatedAt
  deletedAt
  schemaVersion
  sourceDeviceId
- Added native migration:
  004_add_balance_sync_ready_local_fields
- Migration preserves existing rows and backfills owner/device/timestamp/schema
  metadata.
- Migration is idempotent.
- Normal list reads hide deleted_at rows.
- Backup/sync-specific reads can include tombstones.
- New balance rows get owner/sync metadata.
- Web balance stubs were kept aligned with the native API shape.
- Account linking now includes balance rows.

Backup behavior completed
- Backup payload schema was bumped to version 2.
- Backup payload now includes:
  balanceTypes
  balanceEntries
  includesBalance: true
- Backup uploads balance tombstones when local tombstones exist.
- Supabase backup upsert order:
  categories
  balance types
  transactions
  balance entries
- Balance upserts use onConflict: user_id,id.
- Repeated backup does not duplicate balance rows.
- Guest mode does not invoke remote backup writes.
- Backup UI copy remains aggregate-only and safe.

Restore behavior completed
- Restore is authenticated-only, manual-only, merge-only, and non-destructive.
- Restore does not replace local data with cloud data.
- Restore applies balance types before balance entries.
- Active remote balance rows use insert-ignore style local merge.
- Repeated restore does not duplicate balance rows.
- Remote balance tombstones soft-delete matching local rows by stable id.
- Orphan remote tombstones do not create visible local rows.
- Settings refreshes transactions, categories, and balance after successful
  restore.
- Restore UI copy remains aggregate-only and safe.

Manual and foreground sync behavior completed
- Existing manual sync pipeline now includes balance pull/apply/push.
- Existing foreground auto-sync hook is preserved and uses the shared sync
  pipeline, now including balance.
- No sync was added on app start, login, session restore, sign out, background,
  or mutations.
- Balance conflict handling follows the existing deterministic LWW style.
- Sync applies balance types before balance entries.
- Sync summaries include aggregate balance counts.
- Old persisted sync summaries with missing balance counts parse as zero.
- Settings refreshes transactions, categories, and balance after successful
  manual sync and successful existing foreground sync.
- Guest mode does not invoke remote sync writes.

Home History behavior completed
- Home History now builds a union feed, not a Transaction list.
- Transaction feed items:
  show category icon/name
  show negative signed amount, for example -20 €
  preserve existing swipe edit/delete behavior
- Balance feed items:
  show balance type name
  use safe fallback name "Balance addition" when type is missing
  show positive signed amount, for example +100 €
  are not swipeable
- Feed items are filtered by the selected period for both transactions and
  balance entries.
- Feed items are sorted newest-first by createdAt.
- Today summary remains transaction-only.
- Add routes to /add-balance.
- Spend routes to /add-transaction.

Settings and safety changes
- Settings backup/restore copy includes aggregate balance counts.
- Settings sync copy remains aggregate-only.
- Settings no longer logs raw caught errors for relevant operations.
- Startup/onboarding raw caught error logs were also tightened to generic
  operation messages.
- UI/logs should not expose raw backend errors, env values, user ids, owner ids,
  localOwnerIds, deviceIds, tokens, or row payloads.
