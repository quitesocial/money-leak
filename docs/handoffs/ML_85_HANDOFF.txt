Implemented Scope

1. Add Type is now a true Add Balance sub-screen state
- In src/features/add-balance/add-balance-screen.tsx, tapping + Add in the Type
  section switches from Add Balance to an Add Type state.
- Add Type state now shows:
  - the shared custom back header,
  - title: Add Type,
  - Name field,
  - sticky footer CTA: Save Type.
- Add Type state hides:
  - Amount,
  - Date,
  - Type chips,
  - Save Balance CTA.
- Pressing the header back button while in Add Type returns to Add Balance
  instead of navigating away.
- Returning from Add Type preserves the in-progress Add Balance draft, including
  amount, date, and selected type.
- The previous inline Add Type panel pattern was removed.

2. Type name handling
- AddBalanceScreen now normalizes the type name before calling
  useBalanceStore.addBalanceType.
- Existing normalization behavior remains:
  - trims surrounding whitespace,
  - collapses repeated internal whitespace,
  - blocks empty names,
  - blocks duplicate active type names case-insensitively.
- After successful type creation, the newly created type is auto-selected.
- Save Balance then uses the selected/newly created typeId for the BalanceEntry.

3. Preserved behavior and contracts
- Existing balance type selection still works.
- Edit Balance still reuses AddBalanceScreen and preserves the existing balance
  entry ID.
- BalanceEntry and BalanceType remain separate from Transactions.
- Balance additions were not encoded as Transactions.
- Transaction CSV v1 was not changed.
- No store API, DB schema, remote sync/backup contract, Supabase schema, route,
  tab, package version, app config, or EAS config changes were made.
- No new UI dependencies were added.
- AGENTS.md was not updated because this task did not add a durable project rule.
