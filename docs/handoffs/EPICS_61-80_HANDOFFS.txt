Auth restore behavior after ML-61
---------------------------------
- App startup still calls initializeAuth() from app/_layout.tsx without
  blocking local DB initialization, onboarding, navigation, transactions,
  categories, import, or export.
- If Supabase config/client is unavailable:
  - stale Money Leak display auth is cleared;
  - Supabase auth storage boundary is cleared;
  - app enters guest/local mode.
- If Supabase has no session:
  - stale Money Leak display auth is cleared;
  - Supabase auth storage boundary is cleared;
  - app enters guest/local mode.
- If Supabase session restores successfully:
  - Supabase user is normalized into Money Leak AuthSession/AuthUser;
  - only token-free Money Leak AuthSession is persisted;
  - auth store enters authenticated state.
- If Supabase restore is corrupted, revoked, invalid, or fails:
  - Supabase and Money Leak auth storage are cleared;
  - auth store falls back to guest/local mode through the existing safe
    recoverable session_restore_failed path;
  - no crash is expected.
- If getUser validation fails with a retryable network error:
  - an otherwise valid cached Supabase session is kept;
  - token-free Money Leak display session is refreshed from cached session data;
  - offline startup remains locally usable.

Sign Out behavior after ML-61
-----------------------------
- Sign Out calls Supabase auth.signOut({ scope: 'local' }) when a client is
  available.
- Sign Out then clears:
  - Supabase SecureStore auth session boundary;
  - Money Leak token-free display auth session.
- Sign Out does not delete or mutate transactions/categories.
- Sign Out does not change ownerId, localOwnerId, device identity, or local data
  ownership.
- Sign Out does not trigger backup/sync/linking behavior.

Settings/UI behavior after ML-61
--------------------------------
- Continue with Google remains gated exactly by:
  featureFlags.googleAuthEnabled && googleAuthAdapter.isEnabled
- Guest copy remains safe when config is unavailable.
- No Account diagnostics labels are rendered in production Settings UI.
- No raw env values, URLs, keys, tokens, or secrets are rendered.

- Transaction tombstones compare deletedAt when present.

  deletion wins.

- createdAt is preserved when remote winners update existing local rows.
- Stable IDs remain stable.
- No conflict UI was added.

- Remote active transaction with no matching local row is inserted locally.
- Remote active transaction newer than local row overwrites mutable fields

- Local new/edited transaction is pushed to remote.
- Local transaction tombstone is pushed to remote.
- Remote transaction tombstone soft-deletes a matching local row.
- Remote transaction tombstone with no matching local row is ignored and does
  not create a visible local row.
- Repeated sync does not duplicate rows.
