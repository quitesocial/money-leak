ML-65 Manual Backup MVP Handoff

Summary
- Added authenticated, manual, Settings-triggered remote backup.
- The app remains local-first. Guest/local mode remains permanently available.
- Local data remains the source of truth and is never deleted, restored,
  overwritten, merged, or uploaded automatically by this work.
- No restore UI, restore prompt, automatic backup, session-restore upload,
  background sync, incremental sync, delete account, Apple Sign-In, account
  diagnostics UI, CSV change, navigation change, or visual redesign was added.

App changes
- Added src/lib/sync/supabase-remote-backup-adapter.ts.
- Added src/lib/sync/manual-backup-service.ts.
- Added src/db/backup-status.ts with native and web implementations.
- Enabled featureFlags.backupEnabled.
- Kept restoreEnabled, incrementalSyncEnabled, and appleAuthEnabled disabled.
- Settings now shows a Backup card only for authenticated users when backup is
  enabled.
- Backup button label is "Create backup now" and loading text is
  "Creating backup...".
- Success copy reports uploaded transaction and category counts.
- Failure copy is always "Couldn't create backup. Try again."
- Last successful backup timestamp is stored locally in app_metadata under
  last_successful_backup_at and shown only when available.

Remote backup behavior
- Uses the existing Supabase anon client and persisted auth session.
- Does not use a service role key or service role client.
- Upserts remote_categories first, then remote_transactions.
- Both writes use onConflict: user_id,id.
- Local transaction category is preserved as remote category_id.
- Repeated manual backup is safe and does not create duplicate remote rows.
- Remote adapter and Settings UI do not expose raw backend errors, tokens,
  public env values, account identifiers, owner identifiers, or device
  identifiers.

Manual Supabase verification steps
- Log in through Google.
- Tap Create backup now in Settings.
- Verify rows appear in remote_transactions and remote_categories.
- Run backup twice and confirm there are no duplicate rows by (user_id, id).
- Add a local transaction and run backup again.
- Confirm the new transaction is uploaded and existing rows are upserted, not
  duplicated.
- Sign out and confirm local transactions/categories remain visible and
  unchanged.
- Confirm guest mode has no login wall and no backup action.
- Confirm restore, sync, automatic upload, delete account, Apple Sign-In, CSV
  changes, and navigation changes are absent.
