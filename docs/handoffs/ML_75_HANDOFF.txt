ML-75 Foreground Auto Sync v1 Handoff

Summary
- Added foreground-only incremental sync for authenticated users when the app
  returns from background/inactive to active and the last successful sync is
  missing or older than 15 minutes.
- Kept foreground sync opportunistic and silent: local app usage, guest mode,
  and local transaction/category mutations do not depend on sync availability.
- Bumped the app minor version from 1.17.2 to 1.18.0 because this adds a new
  user-visible sync capability.

What changed
- Added src/lib/sync/use-foreground-auto-sync.ts and mounted it from
  app/_layout.tsx.
- Extended the sync service boundary with isIncrementalSyncInFlight() so
  foreground sync can skip when manual or another sync is already running.
- Added focused foreground AppState tests and a sync-service in-flight accessor
  test.
- Updated Settings sync copy to avoid implying sync is only manual.
- Updated docs/mvp-qa-checklist.md with ML-75 QA coverage.
- Updated package.json and package-lock.json version fields to 1.18.0.

Preserved constraints
- Guest/local mode remains permanently available; no login wall was added.
- Add/edit/delete transaction/category remains local-first and is not blocked
  by missing network or sync failure.
- No sync runs on cold start, login, session restore, sign out, or every local
  mutation.
- Foreground sync uses the existing sync service boundary and never calls
  Supabase remote adapters directly from UI/root lifecycle code.
- Manual Settings Sync now remains unthrottled by the foreground 15-minute
  window and still uses the shared in-flight sync guard.
- CSV v1 header remains exactly
  id,amount,category,isLeak,leakReason,note,createdAt.
- Bottom tabs remain Home, Analytics & Leaks, and Settings.
- Add Transaction and Shame Card remain pushed root Stack screens.
- Settings/sync UI remains safe and does not render raw env, token, provider,
  backend, row payload, device, owner, or user-id diagnostics.
- No service-role/admin Supabase client or key was added to mobile
  code/config/package files.
- No @expo/ui, @expo/ui-swift-ui, BlurView, expo-blur, Liquid Glass, or glass
  styling was added.

Intentionally not implemented
- No background sync.
- No sync on every transaction/category mutation.
- No conflict UI.
- No category tombstone product behavior.
- No CSV v2 or CSV metadata fields.
- No backup/restore redesign.
- No navigation changes.
- No new dependencies.

Owner manual QA checklist
- In guest/local mode, background and foreground the app and confirm no sync UI
  appears and local add/edit/delete stays available.
- Sign in with Google, run Sync now once, then background and foreground the
  app within 15 minutes and confirm no repeated sync batch is observed.
- After 15+ minutes from the last successful sync, background and foreground
  the app and confirm fresh remote changes can appear without tapping Sync now.
- Force a recoverable network/backend sync failure, return the app to
  foreground, and confirm local add/edit/delete still works with no raw
  diagnostics shown.
- Start a manual Sync now, immediately background/foreground the app, and
  confirm only one sync operation runs.
- Tap Sign Out and confirm sync does not run and local data remains visible
  after returning to guest/local mode.
- Add, edit, and delete local transactions/categories and confirm those local
  mutations do not trigger sync until manual sync or a later eligible
  foreground return.
- Recheck CSV import/export, bottom tabs, Add Transaction, Shame Card, Backup,
  Restore, and Delete Account flows.
