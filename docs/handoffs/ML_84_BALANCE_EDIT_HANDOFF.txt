Manual QA

Implemented Scope

1. ML-84 Add Transaction one-page redesign
- /add-transaction was changed from the old route-local wizard into one main
  one-page Add Transaction screen.
- Native header remains hidden for Add Transaction; the screen uses its custom
  in-flow header.
- Layout matches the intended Figma direction:
  - #f7f7f5 background.
  - Centered narrow content.
  - Amount, Date, Type, conditional Reason, Category, Add Category, and sticky
    bottom CTA.
- The bottom CTA label is Save, not Next.
- Save submits directly; it no longer advances through details/leakReason/category
  wizard steps.
- No initial transaction Type is selected by default, preserving validation.
- Reason is hidden until Leak is selected.
- Category is also hidden until the user has selected the needed prior fields:
  - Amount + Normal => Category appears and the screen scrolls to Category.
  - Amount + Leak => Reason appears and the screen scrolls to Reason.
  - Selecting a Leak reason => Category appears and the screen scrolls to Category.
- The reason-to-category scroll issue was fixed.
- Selecting Normal clears any previously selected leak reason.
- Category chips render active, non-archived categories only.
- Category chips show the persisted category icon plus display name.
- Save requires a valid selected active category ID.
- Add Category remains an in-screen sub-flow with existing validation,
  CategoryIconPicker behavior, fallback icon, and safe auto-select by normalized
  created name.
- Amount parsing behavior remains:
  - required,
  - must be > 0,
  - dot decimals or one comma decimal accepted.
- LocalDatePicker is used; confirmed date is saved as
  createdAt: selectedDate.getTime().
- Transaction save payload uses useTransactionsStore.addTransaction with:
  - category as stable category ID only,
  - leakReason only for leaks,
  - note: null,
  - generated transaction ID.
- After successful save, navigation goes router.back() when possible, otherwise
  router.replace('/(tabs)').

2. Version and QA docs
- package.json and package-lock.json were bumped from 1.22.1 to 1.23.0.
- A follow-up patch bump was applied from 1.23.0 to 1.23.1 to retrigger the
  GitHub Actions release/build flow and upload a fresh App Store build.
- Final package/app version for this handoff: 1.23.1.
- app.config.js and eas.json were intentionally not changed.
- docs/mvp-qa-checklist.md received an ML-84 manual QA section.

3. Balance additions / income can now be edited and deleted
- The user noticed income/balance additions were not editable/deletable while
  expense transactions were.
- Balance entries remain a separate domain from expense Transactions.
- Balance additions were not encoded as Transactions.
- Transaction CSV v1 remains unchanged.
- New native balance persistence methods were added:
  - updateBalanceEntry(entry)
  - deleteBalanceEntry(id)
- deleteBalanceEntry performs a soft delete by setting deleted_at and updated_at
  on balance_entries.
- updateBalanceEntry updates amount, type_id, created_at, updated_at, and
  source_device_id for non-deleted balance_entries rows.
- Web balance persistence stubs were updated with matching native-only methods.
- The platform balance barrel exports the new update/delete methods.
- useBalanceStore now exposes:
  - updateBalanceEntry(entry)
  - removeBalanceEntry(id)
- Both methods refresh balanceEntries after persistence and use generic store
  errors.

4. Edit Balance route/screen
- Added route: app/balance/[id]/edit.tsx.
- Added screen: src/features/edit-balance/edit-balance-screen.tsx.
- Added root Stack route registration for balance/[id]/edit with native header
  hidden.
- Edit Balance loads balance entries if needed, handles invalid/missing IDs, and
  routes back to /(tabs) from the missing-state action.
- Edit Balance reuses AddBalanceScreen with:
  - initialEntry,
  - title="Edit Balance",
  - submitLabel="Save Changes",
  - onSubmit wired to useBalanceStore.updateBalanceEntry.
- AddBalanceScreen was generalized to accept optional props:
  - initialEntry,
  - onSubmit,
  - submitLabel,
  - title.
- AddBalanceScreen edit mode keeps the existing balance entry ID stable.
- Add Balance behavior remains the default when no props are passed.

5. Home history balance row actions
- Balance history rows now expose swipe edit/delete affordances, matching expense
  transaction rows.
- Edit balance action pushes /balance/{id}/edit.
- Delete balance action shows confirmation:
  - title: Delete balance addition?
  - message: This will remove this income entry from your balance history.
- Confirming delete calls useBalanceStore.removeBalanceEntry(id).
- Swipe state now tracks open/deleting state for both transaction rows and balance
  entry rows.
- Transaction rows keep their previous edit/delete behavior.
- docs/mvp-qa-checklist.md was updated so Home QA no longer expects balance rows
  to be non-swipeable.

6. Home history amount color
- Balance additions / income rows with + amounts are green.
- Expense rows with - amounts remain black.
- Current implemented colors:
  - + amount: #34c759
  - - amount: #050505
